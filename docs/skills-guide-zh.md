# My Skills 技能详解

## 概览

My Skills 包含 10 个技能，覆盖从需求澄清到代码交付的完整开发流程。
通过 `skills/TRIGGERS.md` 意图匹配表自动触发，无需手动指定。

---

## 技能列表

### 🔍 grill（烤问）

**触发信号**：新功能、行为变更、API/数据变更、领域术语不清

在写代码之前，先检查现有代码和文档，找出还没想清楚的设计分支
（行为、数据、API、权限、错误处理、领域语言），然后一次问一个问题，
每个问题附带推荐答案。至少问 3 个有意义的问题，代码里已能回答的跳过。

问完后根据改动大小选择文档级别：

| 级别 | 文件 | 何时 |
|---|---|---|
| 0 | 无 | 改颜色、修 typo |
| 1 | `docs/agents/notes/YYYY-MM-DD-topic.md` | 单模块小功能 |
| 2 | `docs/agents/specs/YYYY-MM-DD-topic-design.md` | 跨模块、API/数据流 |
| 3 | Level 2 规格 → 路由到 `plan` | 需要排期 |
| ADR | `docs/adr/NNNN-title.md` | 不可逆的重大决定 |

如果发现领域术语模糊，直接更新 `CONTEXT.md`。

**门控**：文档写完后必须等用户批准，不批准不写代码。

---

### 📋 plan（计划）

**触发信号**：需要排步骤、拆任务、多步实现

把 grill 批准的文档变成可执行的实现计划：

1. 拆成有序的行为切片
2. 每个切片标注要改的文件
3. 每个切片附带验证命令
4. 标注依赖、风险、回滚点
5. 写到 `docs/agents/plans/YYYY-MM-DD-topic.md`

工作量大时进一步切成独立 issue 写到 `docs/agents/issues/`，
每个 issue 可以独立 review 和测试。

如果源文档不存在或没批准，踢回 `grill`。

---

### 🔴🟢 tdd（测试驱动开发）

**触发信号**：开始写代码、实现功能、修复 bug（诊断完成后）

严格红绿循环，一次只实现一个行为：

1. 从批准的文档中选一个行为
2. 写一个通过公共接口的测试
3. 跑测试，确认**失败**且原因正确
4. 写最小实现让它通过
5. 跑测试，确认**通过**
6. 绿了之后才能重构
7. 更新计划中对应步骤的状态

**规则**：

- 一次一个行为，不准先写完所有测试再写所有代码
- 测试描述系统「做什么」，不描述内部「怎么做」
- 好的测试名：`user can export profile as json`
- 坏的测试名：`calls exportProfile helper`
- 如果生产代码写在测试之前，停下来重新开始

纯视觉调整、文案修改、格式化不走 TDD，直接做。

---

### 🔬 diagnose（诊断）

**触发信号**：bug、报错、异常、失败、空响应、没看到、没显示、性能退化

先诊断，后修复。不准上来就改代码：

1. 建最快的反馈循环（测试/curl/CLI 脚本）
2. 复现用户的实际症状
3. 写 3–5 个按可能性排序的可证伪假设：
   `如果<原因>，那么<探针>会显示<结果>`
4. 一次验证一个假设
5. 确认根因后转成回归测试
6. 告知根因和修复范围，用户确认后才改
7. 修完重跑反馈循环和回归测试
8. 清理 debug 日志

**停止条件**：

- 建不了反馈循环 → 说明原因，要日志/权限
- 没有测试接缝 → 记录为架构发现，转 `architecture`
- 反复修不好 → 停止打补丁，重新诊断

---

### ✅ finish（完成 + 收尾）

**触发信号**：搞定了、commit、push、PR、merge

两部分：先验证，再做 git 操作。

**验证门控**（不可跳过）：

1. 检查当前 diff
2. 对照批准的文档/计划，确认改动匹配
3. 检查有没有偷改不该改的（scope creep）
4. 跑验证命令，看实际输出
5. 说出真实结果——验证被阻断就直说，不许假装通过

以下都是红旗，不接受：
- 「应该能过」
- 「看起来修好了」
- 「我之前跑过」

**Git 操作**（验证通过后）：

- 检测当前分支
- 提供选项：本地合并 / push + PR / 保留分支 / 丢弃
- PR 前检查重复 PR 和 PR 模板
- 丢弃前要求明确确认

---

### 👀 review（审查反馈处理）

**触发信号**：收到 code review 意见、reviewer 提了修改请求

审查意见是需要评估的输入，不是必须服从的命令：

1. 先读完所有反馈再动手
2. 对照代码库验证 reviewer 的说法
3. 正确的 → 接受并逐条实现
4. 错误的 → 用证据反驳

**该反驳的情况**：

- 建议会破坏现有行为
- Reviewer 漏了项目约束
- 要求的功能是投机性的
- 与已批准的设计文档或 ADR 冲突

改完后走 `finish` 验证。

---

### 🏗 architecture（架构深化）

**触发信号**：耦合严重、模块边界不清、测试接缝差、代码流向混乱

只在当前任务范围内改善架构，不搞大规模无关重构：

1. 从当前文件 zoom out 看相关模块和数据流
2. 找浅模块、隐藏耦合、混合职责、差的测试接缝
3. 提出深模块方案：小接口、深实现、清晰权责
4. 每个建议绑定到当前的功能/bug/可测试性问题

**信号**：

- 调用方需要了解内部细节
- 测试必须 mock 私有协作者
- 一个文件因不相关的原因被修改
- 数据结构变更在多个模块间传播

**门控**：分析可以自由做，结构性代码改动必须先过 `grill` 写文档。

---

### 🤖 subagents（子代理协调）

**触发信号**：用户明确要求并行/子代理/delegation

把大块已批准的工作分给多个代理并行做。严格前置条件：

- 已有批准的文档或计划
- 用户明确同意并行
- 工作可按文件/模块切成不重叠的所有权
- 每个子任务有自己的验证方式

**流程**：

1. 识别主线程关键路径（不委托）
2. 选出可以旁路并行的任务
3. 给每个子代理写委托包：目标、拥有的文件、禁止碰的文件、期望的测试
4. 审查每个子代理返回的 patch
5. 合并后走 `finish` 验证

**铁律**：

- 不因为「功能大」就开子代理
- 不把主线程阻塞的任务委托出去
- 不给两个子代理重叠的写入权限
- 不免审直接接受子代理输出

---

### ⚡ prototype（快速原型）

**触发信号**：试一下、spike、PoC、快速验证

唯一不走完整门控的技能，用来回答一个具体问题：

1. 说清楚原型要回答什么问题
2. 建最简单的可运行东西——单文件脚本、终端程序、最小 UI
3. 跳过文档、完整测试、架构审查
4. 给用户看结果
5. 用户决定：扔掉，还是升级为正式功能走 `grill → plan → tdd`

**规则**：

- 原型默认一次性，没有生产期望
- 控制在一个文件或一个小目录内
- 不为原型重构现有代码
- 文件名要能看出是原型

---

### 🗿 caveman（原始人模式）

**触发信号**：简洁点、fewer tokens、terse

把所有回复压缩成短句和标签，去掉客套废话，但绝不跳过任何实质性流程：

- 允许不完整的句子
- 去掉填充词、礼貌用语、模棱两可
- 代码、命令、错误信息、文件路径保持精确
- 保持到用户说「正常模式」

遇到破坏性操作、安全警告、多步指令时临时展开说清楚。

---

## 流程全景

```
用户说了什么
    ↓
匹配 TRIGGERS.md
    ↓
┌─────────────────────────────────────────┐
│ 新功能 → grill → plan → tdd → finish   │
│ Bug    → diagnose → tdd → finish        │
│ Review → review → finish                │
│ 试一下 → prototype                       │
│ 架构问题 → architecture → grill → ...    │
│ 大任务 → ... → subagents → finish       │
└─────────────────────────────────────────┘
```

## 触发机制

三重保障确保技能被激活：

1. **SKILL.md `description`**：每个技能包含中英文关键词，agent 语义匹配
2. **`skills/TRIGGERS.md`**：意图→技能映射表，通过 SessionStart hook 注入
3. **`CLAUDE.md` / `AGENTS.md`**：指示 agent 在任何编码任务前读取 TRIGGERS.md

不需要单独的路由器技能，agent 直接匹配意图。
