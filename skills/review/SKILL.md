# Review

Review feedback is input to evaluate, not an order to obey blindly.

## Process

1. Read all feedback before acting.
2. Verify claims against the codebase.
3. Accept technically correct feedback.
4. Push back with evidence when feedback is wrong, harmful, or YAGNI.
5. Implement valid feedback one item at a time.
6. Test each meaningful change.
7. Run `finish` verification before claiming the review is addressed.

## Push Back When

- The suggestion breaks existing behavior.
- The reviewer missed project constraints.
- The requested feature is speculative.
- The change conflicts with an approved artifact or ADR.
